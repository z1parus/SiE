# Этап 5 — Устойчивость стриков

## Описание

Сейчас стрик **хрупкий**: один пропущенный день — и серия в ноль (`_streak` рвётся на первом
незалогированном дне). Психологически это контрпродуктивно: после случайного срыва пользователь
теряет мотивацию и часто бросает привычку совсем («since I broke it, why bother»). Лучшие трекеры
(Duolingo, Streaks) дают **прощающие механики**: заморозку стрика, дни отдыха, явное различие
«пропуск» vs «срыв», и подчёркивают **рекордный стрик** и **процент выполнения**, а не только
текущую серию. Этот этап делает стрик устойчивым и менее «всё-или-ничего».

Опирается на schedule-aware стрик из этапа 1.

## Пользовательский сценарий

1. У привычки появляется небольшой запас **«заморозок» (streak freeze)** — например, 1 в неделю.
   Если пользователь пропускает запланированный день, автоматически тратится заморозка, и стрик
   **сохраняется** (с пометкой «спасено заморозкой»).
2. **Дни отдыха (rest day):** пользователь может пометить день как «отдых» (например, болезнь) —
   он не считается срывом и не тратит заморозку.
3. На карточке/в деталях показывается **рекордный стрик** («🔥 24 · рекорд 31») — даже после
   срыва есть гордость за рекорд, что мотивирует «побить».
4. При срыве стрика — мягкий, поддерживающий тон («Серия прервалась — но твой рекорд 31 день. Начнём
   новую?»), без «наказывающей» подачи.
5. (Опц.) **Заморозка по приглашению:** если стрик вот-вот сорвётся вечером, вечернее уведомление
   (этап 3) предупреждает: «Сделай X, чтобы сохранить серию 12 дней».

## Логика и поведение

### Различие состояний дня
Для запланированного дня (этап 1) вводим состояния:
- `done` — выполнено (есть лог / цель достигнута).
- `rest` — день отдыха (явная пометка пользователя) — не рвёт стрик, не тратит заморозку.
- `frozen` — пропущен, но спасён заморозкой — не рвёт стрик, тратит 1 заморозку.
- `missed` — пропущен и не спасён — **рвёт** стрик.

### Хранение
- День отдыха — это лог особого вида. Варианты: (а) поле `kind` на `habit_logs`
  (`'done'|'rest'`), либо (б) отдельная таблица `habit_skips`. Рекомендация: **поле `entry_type`
  на `habit_logs`** (`'done'` default | `'rest'`) — переиспользует UNIQUE `(user,habit,date)` и
  существующую инфраструктуру синка.
- Заморозки: счётчик `freezesAvailable` на `Habit` (или вычисляемый бюджет «1/неделя»). Расход
  заморозки фиксируется как `entry_type='frozen'` лог на пропущенный день (создаётся при пересчёте
  стрика). Рекомендация для MVP: **бюджет-правило** «1 заморозка на 7 запланированных дней»,
  применяемое в расчёте стрика, без отдельного счётчика-состояния (проще, без рассинхрона).

### Расчёт устойчивого стрика
- `resilientStreak(habit, logs, restDays, freezeBudget)`:
  - идти от сегодня назад по запланированным дням;
  - `done`/`rest` → серия продолжается;
  - пропуск → если есть бюджет заморозки в окне → пометить `frozen`, продолжить; иначе разрыв.
- `longestStreak` (этап 4) — считается с теми же правилами.

### Edge cases
- **Антифарм отдыха:** нельзя ретроактивно засыпать прошлое «отдыхами», чтобы накрутить стрик —
  ограничить пометку `rest` сегодняшним/вчерашним днём (как уже сделано для retro-log).
- **Чрезмерные заморозки** обесценивают стрик — держать бюджет скромным (1/неделя), настраиваемо.
- **XP** за `rest`/`frozen` НЕ начисляется (это не выполнение) — только сохранение серии.
- **Взаимодействие с Habit Synergy** — буст целей завязан на стрик; устойчивый стрик не должен
  становиться лазейкой для бесконечного буста без реальных выполнений → буст применяется только при
  фактическом `done` (как сейчас, в `toggleHabit`/`logHabitValue`), а не за `rest`/`frozen`.
- **Обратная совместимость** — без пометок отдыха и при нулевом бюджете заморозок поведение
  идентично текущему (серия рвётся на пропуске).

### Схема данных

```sql
ALTER TABLE public.habit_logs
  ADD COLUMN IF NOT EXISTS entry_type text NOT NULL DEFAULT 'done';  -- 'done'|'rest'
  -- 'frozen' — вычисляемое состояние при расчёте стрика, можно не материализовать
```

## Открытые вопросы

1. **Бюджет заморозок** — фиксированное «1/неделя» или зарабатываемый (например, за стрик 7+)?
   Рекомендация: MVP — фиксированный 1/неделя; «зарабатываемые заморозки» как геймификационный
   крючок позже.
2. **Материализовать `frozen`** как лог или считать на лету? Рекомендация: считать на лету
   (меньше состояний, меньше рассинхрона).
3. **Показывать ли заморозки явно** (иконка «❄ осталось 1») или прозрачно спасать? Рекомендация:
   показывать — прозрачность усиливает доверие и делает механику «игровой».
