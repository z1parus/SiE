# Этап 3 — Напоминания и уведомления

## Описание

Напоминания в модуле привычек **полностью отсутствуют** — ни полей в модели, ни сервиса. При этом
напоминание — это **главный драйвер приживаемости** привычки: исследования формирования поведения
(cue → routine → reward) ставят триггер в основу. Этот этап добавляет привычкам время напоминания
(с учётом расписания из этапа 1) и опциональную «вечернюю сводку незакрытых привычек». Сервис
уведомлений — **общий** с roadmap планирования (там этап 2 вводит `NotificationService`); здесь мы
его переиспользуем, а не дублируем.

## Пользовательский сценарий

1. В редакторе привычки — секция **«Напоминание»**: тумблер + выбор времени (например, 07:30).
   Напоминание срабатывает **только в запланированные дни** (учёт `schedule` из этапа 1).
2. Утром приходит push: «💧 Пора: Пить воду». Тап → открывает экран привычек (или сразу деталь
   привычки для быстрого лога).
3. (Опц.) **Вечерняя сводка** в заданное время: «Сегодня осталось 2 привычки: Чтение, Растяжка».
4. При выполнении привычки в этот день её сегодняшнее напоминание больше не имеет смысла — оно либо
   уже сработало, либо отменяется (для «ещё не сработавших»).
5. В системных настройках разрешение отклонено → бережно показываем статус и ссылку в настройки ОС.

## Логика и поведение

### Поля и сервис
- `Habit` получает `reminderTime: String?` (`"HH:mm"`, null = выкл). (Опц. в будущем — несколько
  напоминаний; для MVP одно.)
- Используем `NotificationService` (из roadmap планирования). Если планирование ещё не внедрило его —
  этот этап создаёт сервис (он общий). Методы, нужные привычкам:
  `scheduleHabitReminder(habit)`, `cancelHabitReminder(habitId)`, `scheduleHabitDigest(timeOfDay)`.
- **Детерминированные ID** уведомлений — хэш от `habitId` (+ день недели для weekly-расписаний),
  чтобы перепланирование заменяло, а не плодило.

### Планирование с учётом расписания
- Для `daily` — ежедневное повторяющееся уведомление на `reminderTime`.
- Для `weekdays:1,3,5` — отдельные еженедельные уведомления на выбранные дни в `reminderTime`.
- Для `weekly:N` / `interval:N` — «мягкое» ежедневное напоминание в дни, когда привычка ещё может
  быть выполнена для закрытия цели недели/интервала (или просто ежедневное с текстом-подсказкой —
  компромисс для local-notifications без сервера, см. открытые вопросы).
- Точки внедрения — в `addHabit`/`updateHabit`/`deleteHabit`/`archiveHabit` (планировать/отменять).

### Вечерняя сводка
- Повторяющееся уведомление; контент динамический («осталось N привычек») — пересобирать при
  старте приложения на ближайший вечер по актуальному состоянию (как в планировании).

### Edge cases
- **Разрешение отклонено** — не навязывать; статус в настройках.
- **Время в прошлом сегодня** — первое срабатывание завтра (для daily).
- **iOS-лимит 64 pending** — приоритизировать ближайшие; weekly раскладывается на 7 слотов макс.
- **Архивация/удаление** — отменять напоминания.
- **Смена расписания/времени** — отменить и перепланировать.
- **Часовые пояса/переход на летнее время** — через пакет `timezone` (как в планировании).

### Схема данных

```sql
ALTER TABLE public.habits
  ADD COLUMN IF NOT EXISTS reminder_time text;  -- 'HH:mm' | NULL
```

Настройка времени вечерней сводки — per-device (SharedPreferences), без серверной миграции.

## Открытые вопросы

1. **Local-only или серверные push?** Рекомендация: **local notifications** (офлайн, без бэкенда),
   единообразно с планированием. Серверные push (например, «друг выполнил привычку» для
   accountability) — отдельный будущий этап.
2. **Несколько напоминаний на привычку** (утро+вечер)? Рекомендация: MVP — одно; модель оставить
   расширяемой (строка → список в будущем).
3. **Поведение для `weekly:N`** — ежедневная мягкая подсказка или умный расчёт «осталось закрыть»?
   Рекомендация: MVP — ежедневная подсказка в незакрытые дни недели; умный расчёт позже.
