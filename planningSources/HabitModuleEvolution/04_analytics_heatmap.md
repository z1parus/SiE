# Этап 4 — Аналитика и тепловая карта

## Описание

Сейчас «аналитика» привычки — это три чипа (STREAK / СЕГОДНЯ / ВСЕГО) и таймлайн заметок в
`HabitDetailScreen`. Этого мало, чтобы пользователь **увидел паттерны** своего поведения и получал
мотивацию от прогресса. Данные уже есть (логи за 366 дней грузятся в `_load`), но не визуализируются.
Этот этап добавляет **тепловую карту года** (в стиле GitHub-contributions), процент выполнения,
«лучший день недели», тренды и сводный дашборд по всем привычкам. С типами (этап 2) heatmap
показывает не только факт, но и интенсивность (значение).

## Пользовательский сценарий

1. В деталях привычки под чипами — **тепловая карта** последних ~12 недель / года: сетка дней,
   цвет = выполнено/интенсивность. Видно «полосы» дисциплины и провалы.
2. Карточки инсайтов: «**Выполнение: 78%** за 30 дней», «**Лучший день: среда** (92%)»,
   «**Худший: воскресенье**», «**Рекордный стрик: 24 дня**».
3. Для количественных привычек (этап 2) — линия динамики значения («ср. 6.4 стакана/день, тренд ↑»).
4. Отдельный экран **«Обзор привычек»** (дашборд): общий completion-rate за неделю/месяц,
   тепловая карта «совокупной дисциплины» по всем привычкам, топ и проседающие привычки.

## Логика и поведение

### Вычисления (чистые функции, новый `habit_analytics.dart`)
- `completionRate(habit, schedule, logDates, window)` — выполнено / запланировано за окно
  (знаменатель — **запланированные** дни из этапа 1, не календарные!).
- `bestWeekday(habit)` / `worstWeekday(habit)` — completion по дню недели.
- `longestStreak(habit)` — рекордный стрик за всю историю (schedule-aware).
- `valueTrend(habit)` — для count/duration: скользящее среднее + направление.
- Всё считается **в памяти** на уже загруженных логах — без новых запросов к БД.

### Тепловая карта
- Лёгкий `CustomPainter` `HabitHeatmap` (консистентно с культурой кастомных пейнтеров SiE —
  тактическая карта, орбы, «нейронные нити»). Никаких тяжёлых chart-библиотек.
- Цвет ячейки: для binary — вкл/выкл оттенком `habit.color`; для count/duration — градиент по
  `value/target` (как «contribution levels»).
- `RepaintBoundary` вокруг heatmap (как рекомендовано в дизайн-доке «Нейронной Матрицы»).

### Дашборд
- Новый провайдер `habitAnalyticsProvider` (или `family` по habitId) поверх `habitsProvider` —
  агрегирует метрики; мемоизация, чтобы не пересчитывать на каждый rebuild.

### Edge cases
- **Мало истории** (< 7 дней) — показывать «копим данные», не вводить в заблуждение процентами.
- **Смена расписания** в прошлом — completion-rate считаем по **текущему** расписанию (упрощение;
  «исторические расписания» — оверкилл). Задокументировать.
- **Архивные привычки** — аналитика доступна в архиве (read-only).
- **Производительность** — год = 366 ячеек на привычку; heatmap дешёвый, но дашборд по многим
  привычкам считать лениво/по требованию.

### Схема данных

Новых таблиц **не требуется** — всё считается из существующих `habit_logs` (+ `value` из этапа 2).
Если в будущем понадобится ускорение для очень больших историй — можно кэшировать `longest_streak`
в `habits`, но для MVP это преждевременно.

## Открытые вопросы

1. **Библиотека графиков** — `fl_chart` или собственный `CustomPainter`? Рекомендация: **свой
   CustomPainter** (тепловая карта тривиальна, тренды — простая линия), сохраняем фирменный вид и
   ноль зависимостей.
2. **Глубина heatmap** — год целиком или скролл по месяцам? Рекомендация: 12–16 недель по умолчанию
   + кнопка «весь год» (год на узком экране нечитаем без скролла).
3. **Кэшировать ли `longest_streak`** в БД? Рекомендация: нет для MVP (считаем на лету), добавить
   при проблемах с производительностью.
