# Этап 1 — Гибкое расписание и частота

## Описание

Сегодня каждая привычка неявно **ежедневная**: `Habit` не имеет поля частоты, а стрик (`_streak`,
`habits_provider.dart:161-169`) считает подряд идущие дни от сегодня, предполагая «надо каждый
день». Это фундаментальное ограничение: «спортзал 3 раза в неделю» или «уборка по воскресеньям»
не выражаются, а стрик несправедливо обнуляется в «непредназначенный» день. Этот этап вводит
**расписание привычки** и делает стрик и прогресс **расписание-зависимыми** (schedule-aware).
Это база для напоминаний (этап 3), аналитики (4) и устойчивости стриков (5).

## Пользовательский сценарий

1. При создании/редактировании привычки появляется секция **«Расписание»**:
   - **Каждый день** (поведение по умолчанию — как сейчас).
   - **По дням недели** — выбор чекбоксами (Пн/Вт/…/Вс).
   - **N раз в неделю** — гибкая цель (например, 3 раза, любые дни).
   - **Каждые N дней** — интервал.
2. На главном экране в режиме «Сегодня» показываются **только привычки, запланированные на
   сегодня** (остальные — в свёрнутой секции «Не сегодня», чтобы не зашумлять).
3. Стрик считается корректно: пропуск **непредназначенного** дня НЕ ломает серию. Для «3 раза в
   неделю» стрик measured по неделям («4 недели подряд цель закрыта»).
4. В недельном виде «непредназначенные» дни визуально приглушены (не выглядят как «провал»).

## Логика и поведение

### Модель расписания
- Новое поле `schedule: String` на `Habit` (default `'daily'`), компактный формат:
  - `daily`
  - `weekdays:1,3,5` (1=Пн … 7=Вс)
  - `weekly:N` (N раз в неделю, любые дни)
  - `interval:N` (каждые N дней от `createdAt`/первого лога)
- Чистые функции в `habit.dart`:
  - `bool isScheduledOn(Habit h, DateTime day)` — нужна ли привычка в этот день.
  - `List<DateTime> scheduledDaysInRange(h, from, to)` — для аналитики/стрика.

### Schedule-aware стрик
- Переписать `_streak` → `scheduleAwareStreak(habit, logDates)`:
  - Для `daily`/`weekdays`/`interval`: идти от сегодня назад **только по запланированным дням**;
    серия рвётся, если запланированный день пропущен (не залогирован). Непредназначенные дни
    пропускаются без разрыва.
  - Для `weekly:N`: считать по ISO-неделям — неделя «зачтена», если выполнено ≥ N раз; стрик =
    число последовательных зачтённых недель (текущая неделя — «в процессе», не рвёт серию до её
    конца).
- «Сегодня ещё не наступил дедлайн дня» — незалогированный сегодняшний запланированный день НЕ
  рвёт стрик (серия «жива» до конца дня).

### Фильтрация списка «Сегодня»
- В `habits_provider`/экране: разбить активные привычки на `dueToday` (`isScheduledOn(now)`) и
  `notToday`. Секция `notToday` свёрнута.

### Edge cases
- **Миграция существующих** привычек → `schedule='daily'` (поведение не меняется). Критично для
  обратной совместимости.
- **`weekdays` без единого дня** — запретить (минимум 1 день в UI).
- **`interval:N`** — точка отсчёта: дата первого лога или `createdAt`; задокументировать (рекоменд.
  — первый лог, иначе «каждые 3 дня» считается от создания, что неинтуитивно).
- **Часовые пояса** — всё в локальной зоне; `day = DateUtils.dateOnly`.
- **Habit Synergy (буст целей)** — мультипликатор стрика теперь берёт schedule-aware стрик; убедиться,
  что `planning_provider.applyHabitBoost` получает корректное значение (поведение бустов сохраняется).
- **Производительность** — `isScheduledOn`/стрик считаются на уже загруженных `logDates` в памяти,
  без доп. запросов.

### Схема данных

```sql
ALTER TABLE public.habits
  ADD COLUMN IF NOT EXISTS schedule text NOT NULL DEFAULT 'daily';
  -- 'daily' | 'weekdays:1,3,5' | 'weekly:3' | 'interval:2'
```

RLS не меняется (наследуется от `habits`).

## Открытые вопросы

1. **Как считать стрик для `weekly:N`** — по неделям (рекоменд.) или «дни выполнения подряд
   игнорируя расписание»? Рекомендация: по зачтённым неделям — это честно отражает цель «3 раза
   в неделю».
2. **Точка отсчёта `interval:N`** — от создания или от первого выполнения? Рекомендация: от первого
   выполнения.
3. **Граница недели** — ISO (Пн–Вс) или по локали пользователя? Рекомендация: ISO для предсказуемости,
   с возможной настройкой позже.
