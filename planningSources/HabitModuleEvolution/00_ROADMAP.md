# Эволюция Модуля Привычек — Roadmap

## Цель документа

Превратить модуль привычек SiE из **надёжного, но «плоского» бинарного трекера** в полноценную
**систему построения поведения**: с гибким расписанием, измеримыми целями (количество/время),
напоминаниями, глубокой аналитикой, устойчивыми стриками и инструментами формирования привычек
(сферы жизни, отказ от вредных, стэкинг). Сейчас модуль умеет одно — «отметил/не отметил сегодня».
Этого недостаточно для реального изменения поведения.

---

## Что уже реализовано (база, на которой строим)

| Область | Статус |
|---|---|
| CRUD привычки (название, описание, цвет, иконка-эмодзи) | ✅ |
| Логирование выполнения (1 запись/день, дата `YYYY-MM-DD`) | ✅ |
| Стрик — последовательные дни от сегодня (`_streak`) | ✅ |
| Журнал: заметка + эмодзи к выполнению (`note`/`emoji`) | ✅ |
| Рутины (утренняя/вечерняя, по 1 на пользователя, drag-reorder) | ✅ |
| Закрепление (pin), архивация/восстановление | ✅ |
| Виды: «Сегодня / Неделя / Всё время» + таймлайн в деталях | ✅ |
| Геймификация: +50 XP, +10 DP за выполнение | ✅ |
| Анти-фарм: UNIQUE `(user,habit,date)` + double-tap guard | ✅ |
| Habit Synergy: связь с целями + буст по стрику (1.0/1.5/2.0×) | ✅ |
| Offline-first (Drift ↔ Supabase), batch-загрузка рутин | ✅ |

**Главный вывод аудита:** модель привычки **одномерна** — `Habit` не имеет ни типа измерения,
ни частоты. Логирование — булево «есть запись на дату». Стрик наивно считает подряд идущие дни,
неявно предполагая «привычка = каждый день». Это блокирует целый класс привычек («пить 2л воды»,
«медитировать 20 мин», «спорт 3 раза в неделю», «бросить курить») и делает стрик несправедливым
для неп-ежедневных привычек.

---

## 8 системных пробелов, которые закрывает roadmap

1. **Нет частоты/расписания.** Любая привычка неявно «ежедневная». Нельзя «Пн/Ср/Пт» или «3 раза
   в неделю». Стрик ломается в «непредназначенный» день, демотивируя.
2. **Только бинарные привычки.** Нет количественных («8 стаканов») и временны́х («20 минут») —
   с целевым значением и частичным прогрессом.
3. **Нет напоминаний.** Главный драйвер удержания привычки полностью отсутствует.
4. **Бедная аналитика.** Только таймлайн + 3 чипа. Нет тепловой карты года, процента выполнения,
   «лучшего дня недели», трендов.
5. **Хрупкий стрик.** Один пропуск — стрик в ноль. Нет «заморозки», «дней отдыха», нет учёта
   рекордного стрика и процента выполнения.
6. **Нет группировки.** Привычки в одном плоском списке; при 15+ становится неуправляемо. Нет
   сфер жизни/категорий.
7. **Только «полезные» привычки.** Нет модели «бросить вредное» (счётчик дней воздержания) —
   принципиально другая ментальная модель.
8. **Высокий порог старта + слабые триггеры.** Нет библиотеки готовых привычек; рутины
   ограничены утром/вечером, нет «стэкинга» (привязки привычки к якорю-триггеру).

### Рекомендуемая последовательность внедрения

**Волна 1 — «Настоящий трекер» (ядро):** Этапы 1 → 2 → 3.
Снимает фундаментальное ограничение «бинарно + ежедневно» и подключает напоминания. После этой
волны модуль конкурентоспособен с Habitica/Streaks/Loop.

**Волна 2 — «Мотивация и инсайт»:** Этапы 4 → 5.
Глубокая аналитика и «прощающие» стрики — то, что удерживает пользователя надолго и снижает
выгорание от срыва серии.

**Волна 3 — «Организация и охват»:** Этапы 6 → 7.
Группировка по сферам + поддержка вредных привычек расширяют аудиторию и применимость.

**Волна 4 — «Снижение трения»:** Этап 8.
Библиотека готовых привычек и стэкинг ускоряют старт и повышают приживаемость через научно
обоснованные триггеры (Atomic Habits).

## Сквозные архитектурные принципы (для всех этапов)

1. **Offline-first без исключений.** Новые поля/таблицы — и в Drift (`app_database.dart`), и в
   Supabase; флаги `synced`/`deletedLocally`; optimistic update первым, сеть — через очередь
   `PendingSyncOps`. Сохранять существующий UNIQUE `(user, habit, completed_at)` как фундамент
   анти-фарма — расширять его, а не ломать.
2. **Schema-bump дисциплинированно.** Каждая Drift-миграция увеличивает `schemaVersion` (текущая
   **17**) с блоком `if (from < N)` и ленивой миграцией данных при необходимости. Точный номер
   версии зависит от порядка реализации (координировать с roadmap планирования, который тоже
   претендует на 18+). Каждая Supabase-миграция — отдельный файл с RLS.
3. **Обратная совместимость.** Существующие бинарные ежедневные привычки обязаны работать без
   изменений: новые поля получают значения по умолчанию, эквивалентные текущему поведению
   (`kind='binary'`, `schedule='daily'`).
4. **Riverpod surgical updates.** Расширения `HabitsState` не должны провоцировать перестроение
   всего списка при изменении одной привычки — использовать `.select()` и index-splice, как сделано
   в оптимизации планирования.
5. **Дизайн-система SiE.** Все экраны — `SieBackground`, `c.flatCard`, токены `sieColorsProvider`,
   `SieMotion.enabled` для анимаций. Сохранять фирменную эстетику «Нейронной Матрицы».
6. **Единая геймификация и анти-фарм.** XP/DP — через существующие RPC (`increment_xp`,
   `add_design_points`); новые типы привычек НЕ должны открывать дыры для фарма (частичное
   выполнение начисляет пропорционально/по факту достижения цели, см. этап 2).
7. **Переиспользование, а не дублирование.** Напоминания используют `NotificationService` из
   roadmap планирования (этап 2 там) — единый сервис на оба модуля. Habit Synergy (буст целей)
   при изменении модели стрика обновляется согласованно (`planning_provider`).

---

## Метрики успеха roadmap

- **Adherence (приживаемость):** медиана completion-rate привычек за 30 дней (этапы 1, 3, 5).
- **Глубина:** доля привычек с типом ≠ binary и с расписанием ≠ daily (этапы 1, 2).
- **Удержание:** доля пользователей, у которых ≥1 привычка живёт >21 дня (этапы 3, 4, 5).
- **Охват:** доля пользователей со сферами жизни / вредными привычками (этапы 6, 7).
