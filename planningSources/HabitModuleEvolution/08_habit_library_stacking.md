# Этап 8 — Библиотека привычек и стэкинг

## Описание

Два рычага приживаемости, которых сейчас нет:

1. **Библиотека готовых привычек.** Привычку создают «с нуля»: придумай название, цвет, иконку,
   тип, расписание. Это трение на старте, особенно для новичка, который ещё не знает «как правильно».
   Курируемая библиотека («Пить воду 8 стаканов», «Медитация 10 мин», «10 000 шагов», «Без сахара»)
   с готовыми типом/единицей/расписанием/иконкой даёт старт в один тап.
2. **Habit Stacking (стэкинг / якоря).** Рутины сейчас ограничены двумя типами (утро/вечер). Но
   мощнейшая техника формирования привычек (Atomic Habits: «After I [anchor], I will [habit]») —
   это **привязка новой привычки к существующему триггеру**. Этап обобщает рутины до **именованных
   стэков** с якорем-триггером и порядком, превращая «утро/вечер» в гибкий конструктор цепочек.

Вместе это снижает порог входа и повышает выживаемость привычек через научно обоснованные триггеры.

## Пользовательский сценарий

### Библиотека
1. Кнопка «+» предлагает: «Своя привычка» / **«Из библиотеки»**.
2. «Из библиотеки» → галерея, сгруппированная по сферам (этап 6), карточки с готовой конфигурацией
   («Вода · 8 стаканов · ежедневно · 💧»). Превью + «Добавить».
3. Тап «Добавить» → создаёт привычку с предзаполненными типом/целью/единицей/расписанием/иконкой;
   пользователь может сразу отредактировать.

### Стэкинг
1. В рутинах/стэках можно создать **именованный стэк** с **якорем-триггером** (текст: «После
   утреннего кофе», «Перед сном», «После тренировки»).
2. В стэк добавляются привычки в нужном порядке (как сейчас в `RoutineEditorScreen`, но без лимита
   «только утро/вечер»).
3. На главном экране стэк показывается карточкой-цепочкой: «☕ После кофе → Медитация → Дневник →
   Витамины», с последовательной отметкой выполнения.
4. (Опц.) Напоминание (этап 3) привязывается к стэку целиком.

## Логика и поведение

### Библиотека
- Источник — таблица `habit_templates` (системные, курируемые) либо статический сид в коде
  (`sie_core`). Рекомендация: **таблица в Supabase** (`is_system=true`) + локальное зеркало, чтобы
  обновлять библиотеку без релиза приложения; офлайн — из зеркала/встроенного сида.
- Поля шаблона: `name`, `area`, `kind`, `targetValue`, `unit`, `step`, `defaultSchedule`, `icon`,
  `color`, `polarity`. По сути — «преднастроенный `Habit`».
- Инстанцирование = создать `Habit` из шаблона (переиспользует существующий `addHabit`).

### Стэкинг (обобщение рутин)
- Снять ограничение «routine_type ∈ {morning,evening}, 1 на пользователя». Расширить
  `habit_routines` до именованных стэков:
  - `name: String` (вместо/в дополнение к `routine_type`),
  - `anchor_cue: String?` (триггер-якорь),
  - снять `UNIQUE(user_id, routine_type)` (или мигрировать существующие morning/evening в стэки с
    соответствующими именами).
- `habit_routine_members` и логика порядка/reorder (`reorderMembers`) — переиспользуются как есть.
- UI `RoutineEditorScreen` обобщается: имя стэка + поле якоря + список привычек (drag-reorder).

### Edge cases
- **Миграция существующих рутин** — текущие morning/evening превратить в стэки с именами
  «Утренняя рутина»/«Вечерняя рутина» и якорями по умолчанию; не потерять members и порядок.
- **Дубликаты из библиотеки** — если такая привычка уже есть, предупредить, но разрешить (вдруг
  намеренно).
- **Лимит стэков** — разумный максимум (например, 8), чтобы не превратить главный экран в хаос.
- **Системные шаблоны** — `is_system=true`, только чтение; «добавить» = инстанцировать к себе.
- **Обратная совместимость** — если оставляем `routine_type` для совместимости, новые стэки имеют
  `routine_type=null`; код терпит оба варианта в переходный период.

### Схема данных

```sql
CREATE TABLE public.habit_templates (
  id               uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  name             text NOT NULL,
  area             text,
  kind             text NOT NULL DEFAULT 'binary',
  target_value     double precision,
  unit             text,
  step             double precision,
  default_schedule text NOT NULL DEFAULT 'daily',
  icon             text,
  color            text,
  polarity         text NOT NULL DEFAULT 'build',
  is_system        boolean NOT NULL DEFAULT true
);
-- RLS select: is_system = true (или user_id = auth.uid() для будущих пользовательских)

ALTER TABLE public.habit_routines
  ADD COLUMN IF NOT EXISTS name       text,
  ADD COLUMN IF NOT EXISTS anchor_cue text;
-- снять/смягчить UNIQUE(user_id, routine_type); routine_type → nullable для новых стэков
-- сид: вставить ~15–20 системных шаблонов по сферам
```

## Открытые вопросы

1. **Библиотека: таблица в Supabase или сид в коде?** Рекомендация: **таблица** (обновляемость без
   релиза) + встроенный fallback-сид для офлайна/первого запуска.
2. **Полностью заменить `routine_type` на именованные стэки или сохранить совместимость?**
   Рекомендация: мигрировать morning/evening в стэки, `routine_type` оставить nullable на
   переходный период, затем удалить в будущей миграции.
3. **Пользовательские шаблоны / «сохранить как шаблон»** (как у целей в планировании)? Рекомендация:
   вне MVP; заложить `user_id`/`is_system=false` на будущее.
4. **Стартовый набор библиотеки** — какие 15–20 привычек включить? Предложение: по сферам — вода,
   шаги, медитация, чтение, сон, спорт, растяжка, дневник, витамины, без сахара, без соцсетей,
   изучение языка, планирование дня, благодарность, холодный душ. Финальный список — за продуктом.
