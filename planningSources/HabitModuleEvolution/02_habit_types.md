# Этап 2 — Типы привычек: количество и время

## Описание

Сейчас привычка **бинарна**: лог — это факт «есть запись на дату» (`LocalHabitLogs` PK
`habitId+userId+completedAt`, без значения). Но огромный класс привычек **измерим**: «выпить 8
стаканов воды», «30 отжиманий», «прочитать 20 страниц» (количество) или «медитировать 20 минут»,
«учить язык 30 мин» (время). Этот этап вводит **типы привычек** с целевым значением и **частичным
прогрессом**: пользователь не просто «отметил», а накапливает значение к цели за день, видит
кольцо/полосу прогресса, а выполнение засчитывается при достижении цели.

Вместе с этапом 1 (расписание) это превращает плоский трекер в полноценный.

## Пользовательский сценарий

1. При создании привычки выбирается **тип**:
   - **Да/Нет** (как сейчас, default).
   - **Количество** — целевое число + единица («8 стаканов», «30 отжиманий»).
   - **Время** — целевая длительность («20 минут»).
2. Карточка количественной привычки показывает **кольцо/полосу прогресса** «5 / 8 стаканов» и
   кнопку **«+1»** (или «+шаг»), а также ввод произвольного значения.
3. Тап «+1» инкрементирует дневное значение; при достижении цели — привычка «загорается»
   выполненной (с обычной XP-анимацией). Можно перевыполнить (8/8 → 10/8), но XP начисляется один
   раз за достижение цели.
4. Временна́я привычка: кнопка «+5 мин» / ручной ввод, либо (опц., этап интеграции) запуск таймера.
5. В деталях привычки график показывает не только «выполнено/нет», но и **накопленное значение**
   по дням (динамика: «в среднем 6.4 стакана/день»).

## Логика и поведение

### Расширение модели
- `Habit` получает:
  - `kind: String` — `'binary'` (default) | `'count'` | `'duration'`.
  - `targetValue: double?` — цель за день (8 стаканов, 20 минут, 30 повторов).
  - `unit: String?` — единица («стак.», «мин», «стр.», «раз»).
  - `step: double?` — шаг быстрой кнопки (default 1; для воды — 1 стакан; для времени — 5 мин).
- `habit_logs` получает **`value`** (накопленное значение за день). Для `binary`: `value` = 1.
  UNIQUE `(user, habit, date)` сохраняется — **одна строка на день**, значение накапливается
  через UPDATE, а не новые строки (важно: анти-фарм не ломается).

### Логирование с накоплением
- Новый метод `logHabitValue(habitId, date, delta)` в `HabitsNotifier`:
  - upsert строки лога: `value = COALESCE(value,0) + delta` (клампить ≥ 0).
  - привычка считается выполненной за день, если `value >= targetValue` (для binary — `value>=1`).
  - **XP/DP начисляются один раз** — в момент перехода `value` через `targetValue` (флаг
    достижения цели за день; повторные инкременты не дают XP). Это ключевая анти-фарм-логика.
- `toggleHabit` (бинарный путь) сохраняется для `kind='binary'`; для count/duration основной путь —
  `logHabitValue`. «Снять» выполнение = обнулить дневное значение (с подтверждением).

### Состояние и стрик
- `HabitsState` дополняется `Map<String, Map<String, double>> logValues` (habitId → date → value),
  чтобы UI показывал прогресс. `logDates` остаётся для «достигнута ли цель» (день попадает в
  `logDates`, только когда `value >= target`).
- Стрик (этап 1) опирается на `logDates` — т.е. на **дни достижения цели**, что корректно: частично
  выполненный день не засчитывается в стрик (можно смягчить в этапе 5).

### Edge cases
- **Обратная совместимость:** все текущие привычки → `kind='binary'`, лог `value=1`. Миграция
  проставляет `value=1` существующим логам.
- **Перевыполнение** — разрешено, XP один раз; в аналитике показывать факт перевыполнения.
- **Изменение `targetValue`** у существующей привычки — пересчитать «достигнут ли день» для
  отображения, но НЕ отзывать уже начисленный XP (честно к пользователю).
- **Время:** хранить в секундах или минутах? Рекомендация — хранить в **секундах** (`value`),
  отображать в минутах; совместимо с будущей интеграцией таймера.
- **Дробные значения** (1.5 л) — `value` как `double`; UI форматирует разумно.

### Схема данных

```sql
ALTER TABLE public.habits
  ADD COLUMN IF NOT EXISTS kind         text NOT NULL DEFAULT 'binary',  -- 'binary'|'count'|'duration'
  ADD COLUMN IF NOT EXISTS target_value double precision,
  ADD COLUMN IF NOT EXISTS unit         text,
  ADD COLUMN IF NOT EXISTS step         double precision;

ALTER TABLE public.habit_logs
  ADD COLUMN IF NOT EXISTS value double precision NOT NULL DEFAULT 1;
-- UNIQUE (user_id, habit_id, completed_at) сохраняется: одна строка/день, value накапливается
```

## Открытые вопросы

1. **XP за частичное выполнение** — только при достижении цели (рекоменд., анти-фарм) или
   пропорционально? Рекомендация: только при достижении цели за день; пропорциональный XP открывает
   фарм мелкими инкрементами.
2. **Хранение времени** — секунды vs минуты? Рекомендация: секунды в `value`, отображение в минутах.
3. **Интеграция с модулем Фокуса** для временны́х привычек (запуск таймера → авто-лог)? Перспективно;
   вынести в отдельный мини-этап (перекликается с focus-интеграцией планирования).
