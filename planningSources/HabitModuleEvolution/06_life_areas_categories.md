# Этап 6 — Сферы жизни и категории

## Описание

Привычки живут в **одном плоском списке**, отсортированном лишь по `isPinned`. При 3–5 привычках
это нормально, но при 12+ список становится неуправляемым, и теряется ощущение **баланса жизни**:
сколько внимания идёт в здоровье, ум, отношения, работу. Этот этап вводит **сферы жизни**
(категории) для группировки привычек и лёгкой аналитики баланса. Концепт-документ планирования уже
оперирует категориями целей (`GoalCategory`), а в roadmap планирования есть «Vision / сферы жизни» —
этот этап логично **переиспользует/согласуется** с той же концепцией сфер, давая сквозную картину
«жизни по направлениям» через привычки и цели одновременно.

## Пользовательский сценарий

1. При создании привычки можно выбрать **сферу**: Здоровье / Ум / Продуктивность / Отношения /
   Финансы / Дух (набор по умолчанию, расширяемый). Необязательно — есть «Без сферы».
2. На главном экране привычки **сгруппированы по сферам** (заголовки секций с иконкой/цветом сферы);
   режим группировки переключается (по сфере / плоский список / по рутине).
3. Мини-аналитика баланса: «На этой неделе: Здоровье 90%, Ум 60%, Отношения 20%» — видно, что
   проседает.
4. (Связь с планированием) Если внедрены сферы целей — общий экран «Жизненный баланс» показывает и
   привычки, и цели по одним сферам.

## Логика и поведение

### Модель
- Минимальный вариант (рекоменд. для MVP): поле `area: String?` на `Habit` со значением из
  предустановленного перечисления (как `GoalCategory`). Без отдельной таблицы — быстро, дёшево,
  согласуется с категориями целей.
- Расширенный вариант (если нужны пользовательские сферы): таблица `life_areas`
  (`id,user_id,name,color,icon`) + `habits.area_id`. Это **общая** сущность с планированием
  (его этап Vision вводит `life_areas`). Если внедряется и там, и тут — делаем **одну** таблицу,
  переиспользуемую обоими модулями. См. открытые вопросы.

### Группировка и баланс
- В `habits_provider`/экране — группировка активных привычек по `area`; пустые сферы скрыты.
- Баланс-метрика: средний completion-rate (этап 4) по привычкам сферы за окно.
- Сохранять выбранный режим группировки в `SharedPreferences` (как порядок веток в Hub).

### Edge cases
- **Обратная совместимость** — существующие привычки → `area=null` («Без сферы»), показываются в
  общей секции; ничего не ломается.
- **Согласованность с целями** — использовать тот же перечень/цвета/иконки, что и `GoalCategory`,
  чтобы «Здоровье» выглядело одинаково в обоих модулях.
- **Слишком много сфер** — ограничить набор по умолчанию 6–7; пользовательские (если введём) —
  с разумным лимитом.

### Схема данных

```sql
-- Вариант MVP (предустановленные сферы, согласованы с GoalCategory)
ALTER TABLE public.habits
  ADD COLUMN IF NOT EXISTS area text;  -- 'health'|'mind'|'productivity'|'relationships'|'finance'|'spirit'|NULL

-- Вариант «общие пользовательские сферы» (если согласуем с Vision планирования)
-- CREATE TABLE public.life_areas (id uuid PK, user_id uuid, name text, color text, icon text);
-- ALTER TABLE public.habits ADD COLUMN area_id uuid REFERENCES public.life_areas(id) ON DELETE SET NULL;
```

## Открытые вопросы

1. **Предустановленные сферы vs пользовательские** (таблица `life_areas`)? Рекомендация: MVP —
   предустановленные (поле `area`, как `GoalCategory`); пользовательские ввести **совместно** с
   этапом Vision планирования, чтобы таблица `life_areas` была одна на оба модуля.
2. **Единый перечень с целями** — сделать общий enum «сфера жизни» в `sie_core`? Рекомендация: да —
   сквозная согласованность критична для будущего экрана «Жизненный баланс».
3. **Дефолтный режим отображения** — по сфере или плоский? Рекомендация: плоский по умолчанию (не
   менять привычное поведение), группировку предлагать как опцию.
