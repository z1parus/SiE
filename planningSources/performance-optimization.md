# Оптимизация производительности SiE — скорость и отзывчивость

## Описание

План повышения скорости, отзывчивости и плавности приложения. Основан на сквозном
аудите кода: экран Тактической карты прочитан целиком (`tactical_map_view.dart`, 2005 строк),
слой провайдеров (`planning_provider.dart` и др.) и слой локальной БД (Drift) проанализированы
отдельно.

Документ разбит на блоки:
1. **Тактическая карта** (приоритетный модуль) — самые тяжёлые проблемы рендеринга.
2. **Слой состояния** (Riverpod-провайдеры) — избыточно широкие перестроения.
3. **Слой БД** (Drift/SQLite) — N+1, отсутствие индексов, работа в main isolate.
4. **Сквозные улучшения** по экранам.
5. **Приоритизация и поэтапный план внедрения.**

Главный вывод: узкие места не в «слабом железе», а в архитектуре перестроений.
Сейчас перетаскивание одного узла на карте перестраивает **весь экран целиком** на
каждом событии указателя (60–120 раз/сек), а изменение одной задачи в любой цели
инвалидирует состояние **всех** целей сразу.

## 1. Тактическая карта — приоритет

### 1.1. Полное перестроение Stack на каждом тике драга 🔴 КРИТИЧНО

**Где:** `tactical_map_view.dart:703-709, 772-795, 852-857`

`onPanUpdate` вызывает `setState(() => _positions[id] = ...)` на корневом
`_TacticalMapViewState`. Поскольку дерево карты строится в `build()` этого же State,
**каждое** движение пальца (до 120 событий/сек) заново:
- выполняет `_ensurePositions(goal)` (обход дерева);
- пересобирает список `edges` с нуля (строки 555-580);
- пересоздаёт **все** виджеты-узлы (`for (final sg in _flatSubGoals(...))`, строка 623);
- перезапускает оба `CustomPaint` (грид + рёбра).

Для карты из 40 узлов это означает ~40 пересозданных виджетов + пересчёт всех
безье-кривых **на каждый кадр драга**.

**Решение:** вынести позиции в `ValueNotifier<Map<String,Offset>>` и перерисовывать
только слой узлов/рёбер через `ValueListenableBuilder` / `RepaintBoundary`, **не** дёргая
`setState` всего экрана. Перемещаемый узел оборачивать в отдельный виджет, слушающий
только свою позицию.

```dart
// Вместо setState на весь State:
final _positions = ValueNotifier<Map<String, Offset>>({});

// В onPanUpdate — мутируем notifier, а не State:
onPanUpdate: (d) {
  final next = Map.of(_positions.value);
  next[id] = next[id]! + d.delta / scale;
  _positions.value = next;            // перерисует только слушателей позиций
}

// Слой рёбер и узлов слушает _positions через ValueListenableBuilder,
// корневой build() при драге НЕ вызывается.
```

**Ожидаемый эффект:** падение работы на кадр в ~10–40 раз при драге; плавные 60 fps
вместо «рывков» на больших картах.

### 1.2. `_EdgePainter.shouldRepaint => true` (всегда) 🔴 КРИТИЧНО

**Где:** `tactical_map_view.dart:985-986`

```dart
@override
bool shouldRepaint(_EdgePainter old) => true;   // ← всегда перерисовывает
```

Даже когда ничего не двигается (например, перерисовка из-за смены темы или тика
анимации hover), **все** рёбра пересчитывают безье-кривые и dashed-пути
(`computeMetrics` + `extractPath` в цикле, строки 967-983 — это дорого).

**Решение:** сравнивать реальные входные данные. Список `edges` пересоздаётся каждый
build, поэтому нужно либо сравнивать по значению (сделать `_Edge` сравнимым и сравнивать
списки), либо — лучше — кешировать построенные `Path` и перерисовывать только при
изменении позиций затронутых узлов.

```dart
@override
bool shouldRepaint(_EdgePainter old) =>
    !identical(old.edges, edges) || old.c != c;
```

В связке с п.1.1 (рёбра в отдельном слое, питающемся от `ValueNotifier`) перерисовка
рёбер будет происходить только при реальном движении.

### 1.3. Каскад O(n³) при разрешении коллизий 🔴 КРИТИЧНО

**Где:** `_resolveCollisions` (`275-299`) + `_nodeRadius` (`264-272`)

`_resolveCollisions` — до 30 итераций по всем парам узлов: O(30·n²).
Внутри для **каждой** пары дважды вызывается `_nodeRadius`, а `_nodeRadius`
**на каждом вызове** делает `_flatSubGoals(goal.subGoals)` (полный обход дерева) и
`.expand((sg) => sg.tasks)`:

```dart
double _nodeRadius(String id, Goal goal) {
  if (id == goal.id) return 80.0;
  final allSgs = _flatSubGoals(goal.subGoals);   // ← обход всего дерева на КАЖДЫЙ вызов
  ...
}
```

Итог: 30 · n² · 2 · O(n) = **O(n³)** на каждое завершение драга. Для n=50 это
~7.5 млн операций обхода дерева в одном `onPanEnd` — заметный «фриз» в конце
перетаскивания.

**Решение:**
1. Предрассчитать `Map<String,double> radii` **один раз** до цикла коллизий.
2. Закешировать `_flatSubGoals` результат на время операции (см. п.1.4).
3. Снизить итерации/ввести раннее прерывание по суммарному смещению.

```dart
void _resolveCollisions(String movedId, Goal goal) {
  final radii = {for (final id in _positions.keys) id: _nodeRadius(id, goal)};
  for (int iter = 0; iter < 30; iter++) {
    bool any = false;
    final ids = _positions.keys.toList();
    for (int i = 0; i < ids.length; i++) {
      for (int j = i + 1; j < ids.length; j++) {
        final minD = radii[ids[i]]! + radii[ids[j]]! + 36.0;  // O(1) lookup
        ...
      }
    }
    if (!any) break;
  }
}
```

**Эффект:** O(n³) → O(n²); конец драга перестаёт «заикаться».

### 1.4. `_flatSubGoals` вызывается десятки раз за build 🟠

**Где:** `_flatSubGoals` (`173-181`) дёргается из `build()` (строка 623), из
`addSubGoalEdges`, из `_nodeRadius`, `_nearestSubGoalFor` (`322`), `_tryReparentTask`
(`738`), `_descendantIds` (`343`) и т.д. — каждый раз заново аллоцируя список и
обходя дерево.

**Решение:** вычислять плоский список **один раз** в начале `build()` (и/или при
изменении `goal`) и переиспользовать. Также мемоизировать `_subtreeSize` (сейчас
рекурсивно пересчитывается при каждой радиальной раскладке).

```dart
late List<SubGoal> _flat;     // обновляется когда меняется goal.subGoals
// в build(): _flat = _flatSubGoals(goal.subGoals);  // один раз
```

### 1.5. Корневой `build()` подписан на весь `planningProvider` 🟠

**Где:** `tactical_map_view.dart:537-543`

```dart
final goal = ref.watch(planningProvider).valueOrNull
    ?.goals.where((g) => g.id == _goal.id).firstOrNull ?? widget.goal;
```

Карта подписана на **всё** состояние планирования. Любая мутация в любой другой цели
(или фоновая синхронизация) перестроит карту целиком, хотя нужна одна цель.
В связке с тем, что провайдер заменяет весь `PlanningState` при любой правке (см. §2),
это даёт лишние полные ребилды.

**Решение:** селективная подписка только на нужную цель:

```dart
final goal = ref.watch(planningProvider.select(
  (s) => s.valueOrNull?.goals.firstWhereOrNull((g) => g.id == _goal.id),
)) ?? widget.goal;
```

`select` отсечёт ребилды, когда изменилась не наша цель. Полноценно эффект раскроется
после рефакторинга провайдера на family-по-цели (§2.1).

### 1.6. Отсутствие `RepaintBoundary` вокруг слоёв 🟠

**Где:** `build()` Stack (`595-644`) — грид, рёбра и узлы в одном слое перерисовки.

Грид статичен (меняется только при смене темы), но перерисовывается вместе со всем
остальным. Узлы при драге дёргают перерисовку рёбер и наоборот.

**Решение:** обернуть статичный грид и слой рёбер в `RepaintBoundary`, изолировать
перемещаемый узел. Грид-пейнтер уже корректен (`shouldRepaint => old.c != c`), но без
границы перерисовки это не помогает.

### 1.7. JSON-сериализация позиций в main isolate каждые 300 мс 🟠

**Где:** `_scheduleSave` (`165-171`) → `planning_provider.saveMapPositions` →
`jsonEncode` позиций на UI-потоке.

При активном перетаскивании сейв дебаунсится 300 мс, но `jsonEncode` всё равно идёт
в main isolate. На больших картах (сотни записей позиций) это микрофризы.

**Решение:**
- увеличить дебаунс конца взаимодействия (сохранять на `onPanEnd`, а не по таймеру);
- крупные карты — кодировать через `compute()`/изолят;
- стратегически — хранить позиции отдельной таблицей (см. §3.5), обновляя только
  изменённые строки, без перекодирования всей карты.

### 1.8. Деприкейты `withOpacity` 🟢 (точность + будущая совместимость)

**Где:** массово в `tactical_map_view.dart` (строки 900, 930-937, 1044, 1063, 1124+
и др.) — `flutter analyze` показывает десятки `deprecated_member_use`.

**Решение:** заменить на `withValues(alpha: …)` (точность цвета без потерь, как уже
сделано в Stage 0–5). Чисто механическая замена, снимает шум в анализаторе.

### 1.9. Тяжёлый `InteractiveViewer` с `boundaryMargin: 800` + холст 4800² 🟡

**Где:** `584-593`

Холст 4800×4800 с грид-пейнтером, рисующим 120×120 линий. При больших zoom-out весь
холст в кадре. `_GridPainter` рисует линии по всей `size` без учёта видимой области.

**Решение:** рисовать грид только в видимом прямоугольнике (через
`transformationController` вычислять viewport и клипать), либо заменить грид на
дешёвый тайловый `DecorationImage`/шейдер.

## 2. Слой состояния (Riverpod-провайдеры)

### 2.1. `planningProvider` заменяет ВСЁ состояние при любой правке 🔴

**Где:** `planning_provider.dart` — `addGoal` (396), `deleteGoal` (433),
`updateGoalStatus` (460), `toggleTask` (832), `moveTask` (1319+), `reorderSubGoals`
(1150+), `toggleGoalPin` (1247+) и др.

Каждая мутация делает `state = AsyncData(current.copyWith(goals: [...]))` — новый
объект состояния со всем списком целей. Все слушатели (списки целей, карты, статистика)
перестраиваются, даже если изменилась одна задача в одной цели.

Дополнительно `_updateSubGoalInTree` (23-33) и `buildSubGoalTree` (planning.dart:494)
рекурсивно копируют/пересобирают дерево под-целей при каждой правке —
O(n²) для глубоких деревьев на каждый тоггл/реордер.

**Решение (поэтапно):**
1. **Краткосрочно:** использовать `.select()` на экранах, чтобы отсекать ненужные
   ребилды (см. §1.5).
2. **Среднесрочно:** ввести `planningGoalProvider = FamilyAsyncNotifier<GoalState,
   String>` — правка задачи в цели A перестраивает только слушателей цели A.
3. Кешировать `_allSubGoals`/`_flatSubGoals` в рамках одной операции, не вызывать
   повторно.
4. Геттеры `activeGoals`/`archivedGoals` (planning.dart) — мемоизировать, сейчас
   `.where().toList()` на каждый доступ.

### 2.2. Эталон для подражания — `habits_provider.dart` 🟢

Провайдер привычек уже делает **хирургические** обновления: правит один элемент по
индексу (`[...prev.habits]..[idx] = updated`, строки 387-397), переиспользует
неизменённые коллекции. Это образец, к которому стоит привести planning-провайдер.

## 3. Слой локальной БД (Drift/SQLite)

### 3.1. Запросы выполняются в main isolate 🟠

**Где:** `app_database.dart:830-838` — `driftDatabase(...)` без отдельного изолята.

Тяжёлые загрузки (`_loadFromLocal`) и сериализация идут на UI-потоке. На слабых
устройствах — джанк при старте/обновлении.

**Решение:** вынести БД в фоновой изолят (Drift поддерживает
`NativeDatabase.createInBackground` / isolate-based executor). Это снимает с UI-потока
все запросы и десериализацию.

### 3.2. N+1 при оффлайн-загрузке планирования 🔴

**Где:** `planning_provider.dart:253-358` (`_loadFromLocal`)

```dart
final rawGoals = await db.goalsForUser(userId);     // 1
for (final rg in rawGoals) {
  final rawSubs = await db.subGoalsForGoal(rg.id);   // N
  for (final rs in rawSubs) {
    final rawTasks = await db.tasksForSubGoal(rs.id); // N·M
  }
  final rawMs = await db.milestonesForGoal(rg.id);   // N
  final rawLinks = await db.habitLinksForGoal(rg.id);// N
}
```

10 целей × 5 под-целей ≈ **81 последовательный запрос** при каждом оффлайн-обновлении.

**Решение:** грузить все таблицы целиком одним запросом каждая
(`select(subGoals)..where(userId)`), затем группировать в памяти по `goalId`/`subGoalId`
через `Map`. 5 запросов вместо 80.

### 3.3. Нет индексов на часто-фильтруемых колонках 🟠

**Где:** `app_database.dart` — методы `subGoalsForGoal`, `tasksForSubGoal`,
`goalsForUser`, `habitLogsForUser`, `unsyncedPlanningIds` фильтруют по
`goalId`/`subGoalId`/`userId`/`synced`, но индексов нет (на сервере Supabase индексы
есть, локально — не зеркалированы).

**Решение:** объявить индексы в Drift-схеме (через `@TableIndex` или кастомную
миграцию `createIndex`) на: `goal_id`, `sub_goal_id`, `user_id`, `synced`,
`completed_at`. Линейные сканы → O(log n).

### 3.4. Реордер/балк-апдейты — отдельными запросами 🟠

**Где:** `planning_provider.dart:1158-1160, 1192-1194` — цикл
`await db.updateSubGoalOrderIndex(...)` по одному элементу + столько же вызовов
Supabase.

**Решение:** Drift `batch((b) { ... })` для локали; на сервере — один RPC/`upsert`
массивом. 20 апдейтов → 1 транзакция.

### 3.5. Позиции карты — JSON-блоб целиком 🟡

**Где:** колонка `LocalGoals.mapPositionsJson` + `positionsToJson` (planning.dart).

Каждое сохранение перекодирует **всю** карту; при удалении узлов старые записи позиций
накапливаются (чистка только в памяти, строка 162 карты).

**Решение:** отдельная таблица `map_positions(goal_id, node_id, x, y)` с инкрементальным
upsert изменённых узлов и каскадным удалением. Снимает перекодирование и «протухшие»
позиции.

## 4. Сквозные улучшения по экранам

### 4.1. `ref.watch(sieColorsProvider)` в каждом мелком виджете 🟢
Многие листовые виджеты подписываются на цвета целиком. Тема меняется редко, но
паттерн провоцирует широкие ребилды при любой смене. Рекомендация: для часто
перерисовываемых списков прокидывать `SieColors` параметром (как уже сделано в
узлах карты), а не watch'ить в каждом элементе.

### 4.2. Списки — `ListView`/`itemExtent`/`RepaintBoundary` 🟢
Leaderboard уже использует `itemExtent` и `RepaintBoundary` (хороший паттерн).
Проверить остальные длинные списки (друзья, поиск, привычки) на:
- `ListView.builder` вместо `ListView(children: [...])` (ленивость);
- `itemExtent`/`prototypeItem` где высота фиксирована (ускоряет лейаут и скролл);
- `const`-конструкторы у статичных элементов.

### 4.3. Изображения — кеш и размеры 🟢
Аватары грузятся через `Image.network` (leaderboard, friends) и `CachedNetworkImage`
(customization). Унифицировать на `CachedNetworkImage` с `memCacheWidth/Height` под
реальный размер (44–52 px), чтобы не держать полноразмерные битмапы в памяти.

### 4.4. Анимации — уважать `SieMotion.enabled` 🟢
Hover-анимация карты (`_hoverAnim.repeat`) и прочие повторяющиеся контроллеры —
проверить, что при `disableAnimations` они не крутятся вхолостую (экономия батареи и
кадров).

### 4.5. `const` и вынос статичных поддеревьев 🟢
В тяжёлых `build()` (карта, операции) — максимизировать `const`-конструкторы и выносить
неизменные части в поля/`const`, чтобы Flutter переиспользовал элементы.

## 6. Поэтапный план внедрения

### Этап A — «Быстрые победы на карте» (P0, без смены архитектуры)
Дают наибольший прирост плавности при минимальном риске:
1. **1.3** — предрасчёт радиусов + кеш `_flatSubGoals` в `_resolveCollisions`.
2. **1.2** — корректный `shouldRepaint` у `_EdgePainter`.
3. **1.4** — единый `_flat` на build.
4. **1.5** — `planningProvider.select(...)` для одной цели.
5. **1.6** — `RepaintBoundary` вокруг грида и слоя рёбер.
6. **1.8** — замена `withOpacity` → `withValues` (заодно чистит analyze).

*Проверка:* профилировать драг в DevTools (Timeline) до/после — цель стабильные 60 fps
на карте 40+ узлов.

### Этап B — «Драг без `setState` всего экрана» (P0)
