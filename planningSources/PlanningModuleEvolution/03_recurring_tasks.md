# Этап 3 — Повторяющиеся задачи

## Описание

Сейчас `PlanningTask` — одноразовая: выполнил и всё. Но многие цели держатся на **регулярных
действиях, привязанных к конкретной цели**, которые не дотягивают до полноценной привычки
(«еженедельный отчёт по проекту», «созвон с ментором каждый понедельник», «ревью бюджета 1-го
числа»). Сейчас под такое приходится либо плодить дубликаты задач, либо уходить в трекер привычек
(который живёт отдельно и не привязан к структуре цели). Этап добавляет задачам **правило
повторения**: при выполнении повторяющаяся задача автоматически порождает следующий экземпляр.

Это закрывает разрыв между «одноразовой задачей» и «привычкой», оставаясь внутри дерева цели.

## Пользовательский сценарий

1. При создании/редактировании задачи появляется опция **«Повторять»**: Нет / Ежедневно /
   Еженедельно (выбор дней недели) / Ежемесячно (число) / Каждые N дней.
2. Пользователь делает задачу «Еженедельный отчёт» → повтор «каждый Пн».
3. В понедельник задача появляется в War Room (этап 1) с маленьким значком ↻.
4. Пользователь отмечает её выполненной → задача закрывается, **автоматически создаётся
   следующий экземпляр** на следующий понедельник (с тем же весом, под-целью, правилом).
5. На тактической карте и в списке повторяющаяся задача помечена иконкой ↻.
6. Можно «Завершить серию» (остановить повторение) через контекстное меню — текущий экземпляр
   остаётся, новые не создаются.

## Логика и поведение

### Модель правила повторения
- Новое поле `recurrenceRule: String?` на `PlanningTask` (nullable; `null` = одноразовая).
- Формат — компактный собственный (не полный RFC-5545, чтобы не тащить сложность):
  - `daily`
  - `weekly:1,3,5` (дни недели, 1=Пн … 7=Вс)
  - `monthly:15` (число месяца)
  - `every:N` (каждые N дней)
- Опционально `recurrenceUntil: DateTime?` — дата окончания серии.
- (Опц.) `recurrenceParentId: String?` — связь экземпляров в одну логическую серию для
  аналитики «сколько раз выполнено подряд».

### Порождение следующего экземпляра
- В существующем `toggleTask(...)`: если задача завершается (`isCompleted → true`) и у неё есть
  `recurrenceRule` и (`recurrenceUntil` не прошёл) → вычислить `nextDueDate` от текущего `dueDate`
  (или от `now`, если дедлайна не было) и создать **новую** `PlanningTask`:
  - тот же `subGoalId`, `name`, `weight`, `recurrenceRule`, `recurrenceParentId`;
  - `isCompleted=false`, `dueDate=nextDueDate`, новый `id`, `orderIndex` в конец.
- Вынести расчёт в чистую функцию `DateTime nextOccurrence(String rule, DateTime from)` в
  `planning.dart` (легко тестируется, без side-effects).
- Планирование уведомления для нового экземпляра — через хук из этапа 2.

### Edge cases
- **`weekly` с несколькими днями** — следующий ближайший из списка после `from`.
- **`monthly:31`** в феврале — клампить к последнему дню месяца.
- **Пропуск без выполнения**: если экземпляр просрочен и не выполнен — НЕ порождать второй
  параллельно (иначе накопление). Следующий создаётся только при выполнении текущего. Просроченные
  повторяющиеся задачи попадают в «Просрочено» War Room как обычные.
- **Удаление/заморозка цели** — серия останавливается естественно (новые задачи не порождаются,
  т.к. порождение завязано на выполнение).
- **Подсчёт XP** — каждый выполненный экземпляр даёт `taskXp(weight)` как обычно (это честно:
  повторяющееся действие = повторяющееся усилие). Защита от фарма — вес ограничен, и нельзя
  «закрыть» будущие экземпляры заранее (порождаются по одному).
- **Прогресс цели** — повторяющиеся задачи можно исключать из расчёта `goalProgress`
  (иначе бесконечная задача не даёт цели «завершиться»). Рекомендация: повторяющиеся задачи
  НЕ учитываются в знаменателе прогресса цели — они «двигатель», а не «веха». Это важное
  решение, см. открытые вопросы.

## Схема данных

```sql
-- supabase migration
ALTER TABLE public.planning_tasks
  ADD COLUMN IF NOT EXISTS recurrence_rule      text,        -- 'daily'|'weekly:1,3'|'monthly:15'|'every:N'
  ADD COLUMN IF NOT EXISTS recurrence_until      timestamptz,
  ADD COLUMN IF NOT EXISTS recurrence_parent_id  uuid;
```

RLS не меняется (наследуется от существующих политик `planning_tasks`).

## Открытые вопросы

1. **Учитывать ли повторяющиеся задачи в прогрессе цели?** Рекомендация: **нет** — они движущая
   рутина, а не измеритель завершённости; иначе цель с бесконечной задачей никогда не закроется.
   Показывать их прогресс отдельной метрикой («серия: 6 выполнений подряд»).
2. **Глубина формата повторения** — хватит ли 4 типов (daily/weekly/monthly/every-N)? Рекомендация:
   да для MVP; полный RRULE — оверкилл для мобильного планировщика целей.
3. **Граница с трекером привычек** — не размываем ли мы модули? Позиционирование: привычки =
   образ жизни (без привязки к конкретной цели, своя стрик-механика); повторяющиеся задачи =
   регулярные действия в рамках конкретной миссии. Стоит отразить это в подсказках UI.
