# Этап 7 — Интеграция с Фокусом (план → действие → учёт)

## Описание

В приложении есть модуль **Фокуса** (таймер концентрации, `focusTimerProvider`) и модуль
**Планирования** (задачи), но они живут изолированно. Петля достижения цели разорвана: пользователь
планирует задачи в одном месте, а «работает» (запускает таймер) в другом, и эта работа никак не
привязывается к конкретной задаче/цели. Этот этап **связывает** их: можно запустить фокус-сессию
«над конкретной задачей», а после сессии накопленное время записывается на задачу/цель. Появляется
честный ответ на вопрос «**сколько времени я реально вложил в эту цель**».

Это замыкает цикл **План → Действие → Учёт** и делает время видимым ресурсом.

## Пользовательский сценарий

1. В строке задачи (в `MissionDetailScreen` и в War Room) появляется иконка **▶ «Фокус»**.
2. Тап → запускается модуль Фокуса с **предвыбранной задачей** в заголовке («Фокус: Написать главу 3»).
3. По завершении сессии время (например, 25 мин) записывается как **focus-лог, привязанный к задаче**.
4. На задаче появляется накопитель: «⏱ 1ч 15м вложено». На цели в статистике — «Всего фокус-времени:
   8ч 30м», топ-задачи по времени.
5. (Опц.) Можно отметить задачу выполненной прямо из экрана завершения фокус-сессии.

## Логика и поведение

### Связь сессии с задачей
- Расширить запуск фокус-сессии необязательным контекстом: `focusTimerProvider.start(..., taskRef)`
  где `taskRef = { taskId, goalId }` (nullable — обычный фокус без привязки работает как раньше).
- По завершении сессии (там, где Фокус уже пишет свой лог) — если есть `taskRef`, дополнительно
  записать `focus_logs.task_id`/`goal_id` (или отдельную связь — см. схему).

### Учёт времени
- Агрегаты «время на задачу/цель» считаются из focus-логов:
  - `focusSecondsForTask(taskId)`, `focusSecondsForGoal(goalId)` — методы в Drift (sum по логам).
- Отображать на задаче (компактно) и в `GoalStatsScreen` (карточка «Время в фокусе» + топ-задачи).

### Edge cases
- **Сессия без привязки** — поведение Фокуса не меняется (обратная совместимость; `taskRef=null`).
- **Задачу удалили/перенесли** после сессии — focus-лог остаётся на `goalId` (время вложено в цель),
  `taskId` может стать «осиротевшим» → агрегат по цели всё равно корректен. Использовать
  `ON DELETE SET NULL` для `task_id`.
- **Прерванная сессия** — учитывать по правилам самого модуля Фокуса (частичное время, если он так
  считает). Не дублировать логику — переиспользовать то, что Фокус уже пишет.
- **Антифарм** — XP за фокус начисляет модуль Фокуса по своим правилам; этап НЕ вводит новый XP,
  только привязку и учёт времени (чтобы не плодить дубль-награды).

### Связь с аналитикой (этап 5)
- Фокус-время — отличный вторичный сигнал импульса: цель, в которую недавно вкладывали время,
  «живее». Можно учитывать в `momentumState` как дополнительный фактор (опционально).

## Схема данных

Зависит от текущей структуры логов Фокуса (нужно свериться с реализацией модуля Фокуса). Базовый
вариант — расширить существующую таблицу focus-сессий:

```sql
ALTER TABLE public.focus_sessions   -- имя уточнить по факту
  ADD COLUMN IF NOT EXISTS task_id uuid REFERENCES public.planning_tasks(id) ON DELETE SET NULL,
  ADD COLUMN IF NOT EXISTS goal_id uuid REFERENCES public.goals(id)          ON DELETE SET NULL;
CREATE INDEX IF NOT EXISTS idx_focus_goal ON public.focus_sessions(goal_id);
```

Если у Фокуса нет серверной таблицы (только локальные логи) — добавить связь только в Drift и
агрегировать локально; время-на-цель остаётся per-device (приемлемо для MVP).

## Открытые вопросы

1. **У Фокуса есть серверная таблица сессий или только локальные логи?** Перед реализацией свериться
   с модулем Фокуса. От этого зависит, кросс-девайсное ли «время на цель».
2. **Начислять ли дополнительный XP** за фокус-над-задачей? Рекомендация: **нет** — избегаем
   двойного начисления; ценность — в видимости вложенного времени, а не в новой награде.
3. **Авто-предложение фокуса** из War Room для приоритетной задачи дня? Приятный, но опциональный
   UX-бонус; можно добавить после базовой связки.
