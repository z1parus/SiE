# Этап 8 — Зависимости задач и критический путь

## Описание

Дерево целей сейчас отражает **иерархию** (часть-целое), но не **последовательность** («сначала
A, потом B»). В реальных проектах задачи зависят друг от друга: нельзя «запустить рекламу», пока не
«готов лендинг». Этот этап вводит зависимости «**блокируется → разблокирует**» между задачами
(и под-целями), что даёт три ценности:
1. **Фокус на выполнимом** — War Room и карта подсвечивают только «готовые к старту» (разблокированные)
   задачи, скрывая преждевременный шум.
2. **Критический путь** — визуализация цепочки, определяющей минимальный срок цели.
3. **Честный прогноз** (вместе с этапом 5) — учёт того, что заблокированные задачи нельзя делать сейчас.

Это «продвинутый» этап для пользователей со сложными целями; для простых целей он невидим и ничего
не усложняет.

## Пользовательский сценарий

1. В редакторе задачи — опция «**Зависит от…**» → выбор одной или нескольких задач той же цели,
   которые должны быть выполнены прежде.
2. Заблокированная задача в списке/карте показана приглушённо с замком 🔒 и подсказкой «Ждёт: B, C».
3. Как только все её предшественники выполнены — задача «разблокируется» (визуальный акцент,
   опц. уведомление «Готово к старту: …»).
4. В War Room (этап 1) по умолчанию показываются только **разблокированные** задачи (тумблер
   «показать заблокированные»).
5. На тактической карте зависимости рисуются **особым типом ребра** (пунктир/стрелка), отличным от
   иерархических связей; критический путь подсвечивается.

## Логика и поведение

### Модель зависимостей
- Новая таблица связей `task_dependencies`: `taskId` (зависимая) → `dependsOnTaskId` (предшественник).
  Many-to-many (у задачи может быть несколько предшественников).
- Только в пределах **одной цели** (запрет кросс-целевых зависимостей в MVP — проще и безопаснее).

### Вычисления (чистые функции)
- `isTaskUnblocked(task, allTasksById)` → все `dependsOn` выполнены.
- `readyTasks(goal)` → невыполненные и разблокированные (для War Room/подсветки).
- `detectCycle(...)` → **запрет циклов** при добавлении зависимости (A→B→A). Обязательная валидация
  на этапе создания связи (DFS).
- `criticalPath(goal)` → самая «длинная» по числу/весу цепочка зависимостей до листьев без
  предшественников. Для визуализации; алгоритм — топологическая сортировка + longest path в DAG.

### Edge cases
- **Цикл** — отклонить создание зависимости с понятным сообщением («Это создаст замкнутый круг»).
- **Удаление предшественника** — каскадно удалить зависимости, ссылающиеся на него (`ON DELETE CASCADE`),
  зависимая задача авто-разблокируется.
- **Перенос задачи в другую под-цель** (существующий `moveTask`) — зависимости в пределах цели
  сохраняются (привязка к цели, не к под-цели).
- **Повторяющиеся задачи** (этап 3) — зависимости копируются на новый экземпляр? Рекомендация: нет,
  повторяющиеся обычно независимы; не копировать (упрощает).
- **Производительность карты** — рёбра зависимостей добавляются в существующий слой рёбер
  `TacticalMapView`; учесть в оптимизированном `_repaint`-слое (не в корневом build).

## Схема данных

```sql
CREATE TABLE public.task_dependencies (
  task_id            uuid NOT NULL REFERENCES public.planning_tasks(id) ON DELETE CASCADE,
  depends_on_task_id uuid NOT NULL REFERENCES public.planning_tasks(id) ON DELETE CASCADE,
  created_at         timestamptz NOT NULL DEFAULT now(),
  PRIMARY KEY (task_id, depends_on_task_id),
  CHECK (task_id <> depends_on_task_id)        -- защита от само-зависимости
);
CREATE INDEX idx_task_deps_depends ON public.task_dependencies(depends_on_task_id);
-- RLS: доступ если пользователь owner/editor цели обеих задач (через join planning_tasks→sub_goals→goals)
-- Защита от циклов — на клиенте (DFS) + опц. триггер-проверка на сервере
```

## Открытые вопросы

1. **Разрешать ли зависимости между под-целями**, а не только задачами? Рекомендация: в MVP только
   задачи (под-цель «завершена» = все её задачи; зависимость от под-цели = зависимость от её
   последней задачи — усложняет). Расширить позже.
2. **Серверная защита от циклов** (триггер) или только клиентская? Рекомендация: клиентский DFS
   обязателен; серверный триггер — желателен как страховка, но не блокер для MVP.
3. **Подсветка критического пути** — включать сразу или как отдельную «pro»-фичу? Рекомендация:
   базовые блокировки/разблокировки — в MVP этапа; красивую визуализацию критического пути — вторым
   подшагом.
