# Этап 1 — War Room: единая повестка дня

## Описание

«War Room» — это **операционный центр** модуля планирования: единый экран, который собирает
задачи, вехи и дедлайны со **всех** активных целей и раскладывает их по временной оси
(Просрочено / Сегодня / Завтра / На неделе). Сейчас задачи «заперты» внутри каждой цели —
чтобы понять «что делать сегодня», пользователь обходит каждую миссию вручную. War Room
становится «парадным входом» модуля и превращает приложение из справочника в ежедневный
инструмент.

Это самый высокоприоритетный этап: он не требует новых сущностей в БД (работает поверх
существующих задач с `dueDate`), но даёт максимальный прирост ежедневной ценности.

## Пользовательский сценарий

1. Пользователь открывает раздел Планирования → видит вкладку/кнопку **«War Room»** (или это
   стартовый экран модуля с переключателем на список целей).
2. Вверху — компактная сводка дня: «**4 задачи на сегодня · 1 просрочена · ближайшая веха через 3 дня**».
3. Ниже — секции:
   - 🔴 **Просрочено** (overdue) — задачи с `dueDate < сегодня`, не выполненные. Красный акцент.
   - ⭐ **Сегодня** — `dueDate == сегодня`.
   - **Завтра**.
   - **На этой неделе** — до конца недели.
   - 📍 **Вехи на горизонте** — ближайшие milestone с `targetDate`.
4. Каждая строка задачи показывает: чекбокс, название, **бейдж цели** (цвет+иконка категории),
   вес (XP), дедлайн. Тап по чекбоксу — мгновенно отмечает выполнение (с XP-анимацией), строка
   уезжает в «выполнено». Тап по строке — открывает `MissionDetailScreen` нужной цели.
5. Свайп по задаче → «Отложить на завтра» / «Снять дедлайн».
6. Если на сегодня всё выполнено — мотивационное пустое состояние («Чисто. Все цели под контролем»).

## Логика и поведение

### Агрегация (новый провайдер `agendaProvider`)
- Источник — существующий `planningProvider` (НЕ новый запрос к БД). Это `Provider`, который
  `watch`-ит `planningProvider` и плоско собирает все задачи из дерева всех **активных** целей.
- Рекурсивный обход `goal.subGoals` (дерево) → собрать все `PlanningTask`, обогатив каждую
  ссылкой на родительскую цель (для бейджа): структура `AgendaItem { task, goal, subGoalName }`.
- Аналогично собрать `Milestone` со всех целей.
- Группировка по «корзинам» относительно `DateTime.now()` (локальная дата, обрезка времени):
  `overdue`, `today`, `tomorrow`, `thisWeek`, `later` (свернута по умолчанию).
- Внутри корзины сортировка: сначала по приоритету цели (desc), затем по весу задачи (desc).
- Скрывать выполненные (если `goal.settings.hideCompletedTasks`) — переиспользовать настройку.

### Edge cases
- Задачи **без** `dueDate` не попадают в повестку (это «беклог» цели) — но добавить опциональную
  секцию «Без срока» в самом низу, свёрнутую.
- Цель в статусе `frozen`/`completed` — её задачи в повестку не идут.
- Задача коллаборативной цели, где роль `viewer` — показывать read-only (чекбокс заблокирован).
- Часовые пояса: всё считаем в локальной TZ устройства; «сегодня» = `DateUtils.dateOnly(now)`.
- Производительность: агрегация мемоизируется (как `late final` в `PlanningState`); пересчёт
  только при изменении `planningProvider`.

### Действия из повестки
- Toggle задачи → вызывает существующий `planningProvider.notifier.toggleTask(...)` (XP уже там).
- «Отложить на завтра» → новый метод `rescheduleTask(taskId, newDueDate)` в `PlanningNotifier`
  (optimistic update + upsert + sync). Это маленькое расширение существующего CRUD.
- Тап по строке → `Navigator.push(MissionDetailScreen(goalId))`, по возможности со скроллом
  к нужной задаче.

### Эскиз провайдера (псевдокод)

```dart
class AgendaItem {
  final PlanningTask task;
  final Goal goal;        // для бейджа: цвет + категория
  final String subGoalName;
}

class AgendaBuckets {
  final List<AgendaItem> overdue, today, tomorrow, thisWeek, later, noDate;
  final List<MilestoneRef> upcomingMilestones;
  int get todayCount => today.length;
  int get overdueCount => overdue.length;
}

final agendaProvider = Provider.autoDispose<AgendaBuckets>((ref) {
  final state = ref.watch(planningProvider).valueOrNull;
  if (state == null) return AgendaBuckets.empty;
  final now = DateUtils.dateOnly(DateTime.now());
  // обход активных целей → плоский список → раскладка по корзинам
});
```

## Схема данных

Новых таблиц **не требуется** — этап работает поверх существующих `planning_tasks.due_date`
и `milestones.target_date`. Единственное расширение CRUD — `rescheduleTask`, меняющий `dueDate`
существующей задачи (поле уже есть и в Drift, и в Supabase).

## UX-детали (как топ-инженер)

- **Прогресс-кольцо дня** вверху: «выполнено сегодня / запланировано сегодня» — даёт чувство
  завершённости (Zeigarnik effect), мотивирует закрыть оставшееся.
- **Цвет-кодирование по цели**, а не по типу: пользователь мгновенно видит «над чем работаю».
- **Не показывать гигантские списки**: секция `later` и `noDate` свёрнуты по умолчанию.
- Лёгкий **haptic** на выполнение (как на карте), плюс существующая XP-анимация.
- Пустое состояние «всё чисто» — позитивное подкрепление, а не унылый пустой экран.

## Открытые вопросы

1. **War Room как стартовый экран модуля или как вкладка?** Рекомендация: сегмент-контрол
   вверху `PlanningScreen` — «Повестка | Цели», с запоминанием последнего выбора.
2. Включать ли задачи **без дедлайна** в отдельную секцию или прятать полностью? Рекомендация:
   свёрнутая секция «Без срока» внизу — не теряем беклог, но не зашумляем.
3. Нужен ли горизонтальный **выбор дня** (мини-календарь-стрип на 7 дней)? Можно отложить до
   этапа 9, в MVP — статичные корзины.
