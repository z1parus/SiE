# Эволюция Модуля Планирования — Roadmap

## Цель документа

Превратить модуль планирования SiE из мощного **хранилища структуры целей** в полноценный
**операционный инструмент достижения целей** — то есть в систему, которая не только хранит
дерево целей, но и ежедневно ведёт пользователя к результату: говорит «что делать сегодня»,
напоминает, измеряет темп, прогнозирует исход и закрывает петлю «план → действие → рефлексия».

---

## Что уже реализовано (база, на которой строим)

| Область | Статус |
|---|---|

| Тактическая карта (узлы, рёбра, drag, сохранение позиций) | ✅ |
| Категории целей (learning/health/project/lifestyle/discipline) | ✅ |
| Геймификация: XP (task/subgoal/milestone/goal), DP, медали Bronze/Silver/Gold | ✅ |
| Связь с привычками + динамический буст по стрику (1.0/1.5/2.0×) | ✅ |
| Коллаборация (viewer/editor, до 10, realtime presence, нотификации) | ✅ |
| AI-декомпозиция цели (Groq/Llama → под-цели+задачи+вехи) | ✅ |
| Stagnation detection + Strategic Advice (текстовые советы) | ✅ |
| Fog of War, закрепление целей, настройки, экран статистики | ✅ |
| Offline-first (Drift ↔ Supabase), batch-загрузка, оптимизация рендера карты | ✅ |

**Главный вывод аудита:** в модуле сильная «ось структуры» (как цель устроена), но почти
отсутствует **«ось времени и исполнения»** (что делать сегодня, успеваю ли я, что измеримо
изменилось). Именно туда направлен этот roadmap.

## 7 системных пробелов, которые закрывает roadmap

1. **Нет единого «сегодня».** Задачи имеют `dueDate`, но нет экрана, который собирает задачи
   со **всех** целей в повестку дня. Пользователь вынужден обходить каждую цель вручную.
2. **Напоминания не работают.** Поле `remindBeforeDeadlineDays` есть в `GoalSettings`, но
   локальные уведомления не запланированы — настройка «висит» без эффекта.
3. **Задачи одноразовые.** Нет повторяющихся задач — для регулярных действий приходится
   плодить дубликаты или уходить в трекер привычек.
4. **Вехи бинарные.** Milestone — только «достигнуто/нет». Нельзя отслеживать числовую
   метрику («80 кг», «$5000 накоплений») с динамикой.
5. **Статистика — снимок.** Нет истории прогресса во времени → нельзя показать темп
   (velocity), прогноз даты завершения, burndown.
6. **Высокий порог старта.** Каждую цель создаём с нуля. Нет шаблонов/чертежей миссий.
7. **Петля «план→действие» разорвана.** Модуль Фокуса (таймер) и задачи не связаны — нельзя
   «поработать над этой задачей» и увидеть накопленное время.

### Рекомендуемая последовательность внедрения

**Волна 1 — «Ежедневный драйвер» (ядро ценности):** Этапы 1 → 2 → 3.
Превращает приложение из справочника в инструмент, который открывают каждый день.

**Волна 2 — «Глубина и интеллект»:** Этапы 4 → 5.
Измеримость и прогноз — отвечают на вопрос «успеваю ли я и что реально изменилось».

**Волна 3 — «Снижение трения и закрытие петли»:** Этапы 6 → 7.
Быстрый старт целей + связь планирования с реальной работой.

**Волна 4 — «Продвинутое и удержание»:** Этапы 8 → 9.
Для опытных пользователей со сложными целями + ритуал, удерживающий вовлечённость.

## Сквозные архитектурные принципы (для всех этапов)

Чтобы новые фичи не разрушили существующую архитектуру, каждый этап обязан следовать паттернам,
уже принятым в кодовой базе:

1. **Offline-first без исключений.** Любая новая сущность: таблица в Drift (`app_database.dart`)
   + таблица в Supabase + поля `synced`/`deletedLocally` + enqueue в очередь `PendingSyncOps`.
   Optimistic update состояния идёт первым, сеть — вторым.
2. **Schema-bump дисциплинированно.** Каждая Drift-миграция увеличивает `schemaVersion`
   (текущая — 17) и добавляет блок `if (from < N)` с `createTable`/ленивой миграцией данных.
   Каждая Supabase-миграция — отдельный файл `supabase/migrations/<timestamp>_*.sql` с RLS.
3. **Riverpod surgical updates.** Новые провайдеры используют `.select()` и index-splice,
   как сделано в `planning_provider.dart` после оптимизации, чтобы избегать широких перестроений.
4. **Переиспользование, а не дублирование.** Повестка дня берёт задачи из существующего
   `planningProvider`; интеграция с Фокусом дёргает существующий `focusTimerProvider`; шаблоны
   повторяют паттерн `meditation_presets_provider`. Новые модули встраиваются в дерево, а не
   создают параллельную вселенную.
5. **Дизайн-система SiE.** Все экраны — на `SieBackground`, `c.flatCard`, токенах
   `sieColorsProvider`, с `SieMotion.enabled` для анимаций. Никаких «голых» Material-виджетов.
6. **Геймификация консистентна.** Новые действия начисляют XP/DP через
   `userProfileProvider.notifier.applyLocalXpDelta()` — единая точка, как в текущем `_awardXp`.

## Метрики успеха roadmap

- **DAU/активация:** доля пользователей, открывающих War Room ежедневно (этапы 1–2).
- **Achievement rate:** доля целей, доходящих до `completed` (этапы 3–7 повышают доводимость).
- **Точность планов:** медиана отклонения фактической даты завершения от прогноза (этап 5).
- **Retention 4-week:** возвраты после еженедельного ревью (этап 9).
