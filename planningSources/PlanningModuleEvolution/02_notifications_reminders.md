# Этап 2 — Уведомления и умные напоминания

## Описание

Сейчас поле `GoalSettings.remindBeforeDeadlineDays` существует, но **ни к чему не подключено** —
локальные уведомления не планируются. Этот этап «оживляет» настройку и добавляет полноценный
слой напоминаний: пользователь получает push-уведомления о приближающихся дедлайнах задач,
вехах, дедлайнах целей и ежедневную утреннюю сводку повестки (из этапа 1). Это ключевой механизм
**возврата** в приложение и **доведения целей до результата** — без напоминаний планы забываются.

## Пользовательский сценарий

1. При первом заходе в модуль планирования (или в настройках) запрашивается разрешение на
   уведомления (бережно, с объяснением пользы — не «в лоб» на старте).
2. Пользователь создаёт задачу с дедлайном «послезавтра» → система автоматически планирует
   напоминание (по умолчанию — утром в день дедлайна + за N дней согласно `remindBeforeDeadlineDays`).
3. Утром приходит **дневная сводка**: «🎯 Сегодня по плану: 3 задачи. Ближайшая веха через 2 дня».
   Тап → открывает War Room.
4. За день до дедлайна цели: «⏳ "Сдать проект" — дедлайн завтра. Прогресс 70%».
5. В разделе настроек цели — секция «Напоминания»: тумблеры (дневная сводка вкл/выкл, напоминать
   за N дней, время утренней сводки).
6. При выполнении задачи/изменении дедлайна — связанное уведомление **отменяется/перепланируется**.

## Логика и поведение

### Сервис уведомлений
- Новый `NotificationService` (`packages/sie_core/lib/src/services/notification_service.dart`)
  поверх пакета `flutter_local_notifications` (+ `timezone` для корректного зонирования).
- Методы: `init()`, `requestPermission()`, `scheduleTaskReminder(task, goal)`,
  `scheduleMilestoneReminder(milestone, goal)`, `scheduleGoalDeadline(goal)`,
  `scheduleDailyDigest(timeOfDay)`, `cancel(notificationId)`, `cancelForGoal(goalId)`.
- **Детерминированные ID:** id уведомления = хэш от `taskId`/`milestoneId` (стабильный int),
  чтобы перепланирование заменяло старое, а не плодило дубликаты.

### Когда планировать/отменять
- **Создание задачи с `dueDate`** → запланировать на (`dueDate - remindBeforeDeadlineDays`) и
  на утро дня `dueDate`.
- **Выполнение задачи / удаление / снятие дедлайна** → `cancel(taskNotificationId)`.
- **Изменение `dueDate`** (в т.ч. «Отложить на завтра» из этапа 1) → отменить + перепланировать.
- **Завершение/заморозка цели** → `cancelForGoal(goalId)` (все её задачи и вехи).
- Точки внедрения — внутри существующих методов `PlanningNotifier`: `addTask`, `toggleTask`,
  `deleteTask`, `rescheduleTask`, `addMilestone`, `completeMilestone`, `updateGoalStatus`.

### Дневная сводка
- Повторяющееся уведомление на заданное время (по умолчанию 08:00). Текст формируется при
  срабатывании из снимка повестки. Поскольку контент динамичен, использовать периодический
  ре-scheduling: при каждом запуске приложения пересчитывать «на завтра» сводку с актуальными
  числами (зрелый компромисс для local-notifications без сервера).
- Альтернатива (для будущего): серверные push через Supabase Edge Function + FCM — вынесено в
  «Открытые вопросы».

### Edge cases
- **Разрешение отклонено** → фичу не навязываем, в настройках показываем «Уведомления выключены
  в системе» со ссылкой на системные настройки.
- **Дата в прошлом** → не планируем (иначе мгновенный спам).
- **Массовая правка** (AI-декомпозиция добавила 30 задач) → планировать только задачи с
  дедлайном; батчить вызовы, не блокировать UI (выполнять после возврата управления).
- **iOS-лимит 64 pending notification** — приоритизировать: ближайшие N дедлайнов + дневная
  сводка; дальние планировать лениво при открытии приложения (rolling window).
- **Часовые пояса/перевод времени** — через пакет `timezone`, локальная зона устройства.

## Схема данных

Новых **серверных** таблиц не требуется. Локально нужно хранить соответствие
«сущность → notificationId» только если выбран хэш-подход без хранения (рекомендуется — хэш,
хранение не нужно). Опционально — глобальные настройки уведомлений в существующем механизме
`SharedPreferences`/настроек приложения (не в Supabase), т.к. это per-device.

Если выносим `dailyDigestEnabled`/`digestHour` на уровень аккаунта — добавить 2 поля в
`public.profiles` (+ Drift-зеркало). Рекомендация для MVP: **per-device через SharedPreferences**,
без миграции БД.

## Открытые вопросы

1. **Local-only или серверные push (FCM)?** Рекомендация: начать с **local notifications**
   (работают офлайн, без бэкенда). Серверные push для напоминаний коллаборантам («партнёр
   выполнил задачу») — отдельный будущий этап.
2. **Где хранить настройки напоминаний** — per-device (SharedPreferences) или per-account
   (profiles)? Рекомендация: per-device для MVP; синхронизация позже при необходимости.
3. **Время дневной сводки** — фиксированное 08:00 или выбираемое? Рекомендация: выбираемое, но
   с дефолтом 08:00.
4. Нужны ли **«умные» напоминания о стагнации** (этап интегрируется с `isGoalFatigued`)?
   «Цель X не двигалась 7 дней» — мощно, но требует осторожности с частотой. Предлагаю включить
   как опциональный тумблер «Подталкивать при застое», по умолчанию выкл.
