## Описание

Редизайн `public_profile_screen.dart` (982 строки) поверх `ProfileHeroCard`
(этап 1). Экран показывает чужого оперативника: hero, кнопки дружбы, статистика,
достижения, медали. Имеет несколько P1-проблем из `UX-Audit §5.3`: соц-кнопки
без loading-состояния, удаление друга без подтверждения, сетка ачивок 6 колонок
(тач-таргеты ~35px), back-кнопка сливается с фоном, hero-изображение без кэша.

## Пользовательский сценарий

Пользователь открывает чужой профиль (из поиска, лидерборда, списка друзей).
Видит крупный hero оппонента с его фоном и узором (его кастомизация — часть
его «лица»). Понятная кнопка действия дружбы с явным состоянием загрузки.
Ниже — статистика (концентрация, циклы, ранг), достижения и медали. Всё
консистентно с личным профилем, но read-only + соц-взаимодействие.

## Логика и поведение

### Hero и навигация

- **ProfileHeroCard** (этап 1, `avatarSize: 96`). Узор/фон берутся из
  оснащённых ассетов **просматриваемого** пользователя (требует публичной
  отдачи `equipped_pattern_id`, см. этап 4 / открытые вопросы).
- **Back-кнопка** (UX-Audit 5.3 #4): обернуть в полупрозрачную круглую подложку
  (`_GlassCircleButton`-стиль из личного профиля) — не сливается с hero.
- **Hero-изображение** (`Image.network`, UX-Audit 5.3 #5) → `CachedNetworkImage`
  (решается в этапе 1 внутри `ProfileHeroCard`).

### Соц-кнопки `_FriendActionSection` (831–933)

- Состояния дружбы: «Добавить» / «Отменить запрос» / «Принять» / «Отклонить» /
  «Удалить».
- **Loading-состояние** (UX-Audit 5.3 #1, P1): на время RPC — спиннер внутри
  кнопки + `disabled`, чтобы исключить двойные тапы. Паттерн уже реализован в
  Interface Hub (`_buyingId` + `_ActionButton`) — переиспользовать подход.
- **Подтверждение удаления друга** (UX-Audit 5.3 #2, P1 / 0.2): через общий
  `confirmDestructive(...)` (как в friends_list), с выделением имени.
- Хаптика: `SieHaptics.success()` на принятие заявки, `selection()` на отправку.
- `_SocialBtn` (935–981) — увеличить тач-зону, добавить `Semantics(button:true)`.

### Статистика `_StatsRow` (435–480)

- 3 карточки: Концентрация (время фокуса), Циклы (привычки), Ранг (уровень).
- Привести к виду `_StatsStrip` из личного профиля (этап 2) — единый компонент
  для обоих экранов. Цветовая дифференциация карточек (UX-Audit 5.9 #7 как
  паттерн) — иконка+акцент на каждую.

### Достижения `_AchievementsSection` (562–621)

- Сейчас **6 колонок** (~35px, UX-Audit 5.3 #3, P1) — слишком плотно. Снизить
  до **3–4 колонок**, как в личном профиле (единый `AchievementBadge`,
  адаптивный размер). Тап → `_AchievementSheet` с feedback.

### Медали `_PublicMedalsSection` (792–827)

- Сейчас `Wrap`, медаль 56px. Сохранить `Wrap` (компактно для чужого профиля),
  но выровнять размеры/отступы с витриной личного профиля; тап → деталь.

### Состояния

- Скелетоны (`SieSkeleton`) для `publicStatsProvider` / `publicAchievements` /
  `publicMissionMedals` вместо спиннеров.
- `SieErrorState(onRetry)` для офлайна (UX-Audit 0.4).

### Токены

- Циан-grid `0xFF00C8FF` (строка 232) → `c.info` (UX-Audit 5.3 #6, 0.1).

## Схема данных

Для отображения узора чужого профиля публичный источник
(`get_operative_stats` / `PublicProfile`) должен отдавать
`equipped_pattern_id` (и уже отдаёт `equipped_background_id` для фона —
подтвердить). Деталь — в этапе 4.

## Открытые вопросы

  отдачи equipped-ассетов) или унифицировать read-only оформление. Рекомендую
  показывать кастомизацию владельца — это часть социальной ценности.
- Нужна ли кнопка «пожаловаться / заблокировать» в публичном профиле (за
  рамками этого редизайна, отметить как backlog).
