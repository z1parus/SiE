# Тактическая карта: улучшение управления и вибро-отклики

## Описание

Два связанных улучшения UX тактической карты в модуле Планирования:

1. **Исправление «убегания» элементов** — при перетаскивании элемент «набирает скорость» и уходит из-под пальца, теряется точность управления.
2. **Вибро-отклики и визуальная подсветка** — haptic feedback при взаимодействии с элементами, визуальная индикация смены родителя при перетаскивании.

---

## Пользовательский сценарий

### Сценарий 1: Перетаскивание задачи внутри подцели
Пользователь нажимает на задачу, слегка зажимает её — ощущает лёгкую вибрацию (сигнал: «элемент захвачен»). Перетаскивает в другое место в пределах той же подцели. Палец и элемент движутся синхронно, без ощущения «рывка» или «убегания». При отпускании — короткая вибрация (сигнал: «отпущен»).

### Сценарий 2: Перенос задачи в другую подцель
Пользователь тянет задачу в сторону другой подцели. Как только задача приближается к границам другой подцели (порог ~200px):
- Граница перетаскиваемой задачи подсвечивается акцентным цветом
- Граница целевой подцели пульсирует / меняет цвет (индикация «зоны приёма»)
- Срабатывает вибро-отклик средней силы (сигнал: «попадание в зону нового родителя»)

Пользователь отпускает — задача перемещается в новую подцель, короткая вибрация (сигнал: «задача перенесена»).

Если пользователь отводит задачу от целевой подцели — подсветка и вибрация гасятся.

### Сценарий 3: Тапы и прочие взаимодействия
- Тап по любому элементу (задача, подцель, контрольная точка и т.д.) → лёгкая вибрация.
- Долгий тап (контекстное меню) → средняя вибрация.

## Корень проблемы «убегания»

Файл: `apps/central_hub/lib/screens/tactical_map_view.dart`

**Проблема:** конфликт жестов между `GestureDetector` на узлах и `InteractiveViewer` на холсте.

Текущая реализация:
```dart
onPanUpdate: (d) {
  final scale = _tc.value.getMaxScaleOnAxis();
  setState(() => _positions[task.id] = _positions[task.id]! + d.delta / scale);
},
```

Когда пользователь тянет узел, `GestureDetector` обрабатывает `d.delta / scale` и двигает узел. Одновременно `InteractiveViewer` получает **то же** событие и панорамирует весь холст. Результат: узел движется на `d.delta / scale`, а сам холст сдвигается на `d.delta` — суммарное смещение узла относительно экрана двойное, элемент «убегает».

## Логика и поведение

### Исправление 1: Блокировка InteractiveViewer во время перетаскивания

`InteractiveViewer` имеет параметр `panEnabled`. Нужно динамически выключать его когда `_draggingId != null`.

```dart
InteractiveViewer(
  panEnabled: _draggingId == null,   // ключевое изменение
  scaleEnabled: _draggingId == null, // тоже блокируем зум во время drag
  transformationController: _tc,
  ...
)
```

При этом пересчёт delta не меняется — он уже корректно делится на scale. После отключения конкурирующего жеста холст перестаёт двигаться одновременно с узлом, и синхронность восстанавливается.

### Исправление 2: Захват жеста на элементе

Дополнительно обернуть `GestureDetector` узла в `Listener` с `onPointerDown`, который вызывает `RenderBox.globalToLocal` и явно поглощает событие через `HitTestBehavior.opaque` — это уже установлено в коде, достаточно блокировки InteractiveViewer.

### Вибро-отклики (HapticFeedback)

Импорт уже доступен через `package:flutter/services.dart`.

| Событие | Метод | Интенсивность |
|---|---|---|
| `onPanStart` (начало drag) | `HapticFeedback.lightImpact()` | лёгкая |
| `onPanEnd` (отпускание) | `HapticFeedback.lightImpact()` | лёгкая |
| Успешный reparent (смена родителя) | `HapticFeedback.mediumImpact()` | средняя |
| Вход в зону нового родителя | `HapticFeedback.selectionClick()` | клик |
| `onTap` по любому элементу | `HapticFeedback.selectionClick()` | клик |

Вибрация при «входе в зону нового родителя» должна срабатывать **один раз** при переходе состояния «нет целевого родителя» → «есть целевой родитель» (edge trigger), не повторяясь пока палец удерживается в той же зоне.

---

### Визуальная индикация reparent

Добавляется новое состояние в `_TacticalMapViewState`:

```dart
String? _draggingId;        // уже есть
String? _hoverTargetId;     // новое: id элемента-цели под курсором
```

**Во время `onPanUpdate` для задач и подцелей:**
Выполнять «поиск ближайшего кандидата в родители» (аналог `_tryReparentTask`, но без мутации состояния). Если найден — записывать в `_hoverTargetId` и триггерить вибрацию при изменении.

**Визуал перетаскиваемого элемента:**
- Цвет рамки → акцентный (`c.accent`)
- `borderWidth` 2.5px (сейчас 2.0px при drag)
- Box-shadow усиливается: `blurRadius: 24, spreadRadius: 4`

**Визуал целевого родителя (`_hoverTargetId`):**
- Рамка → акцентный цвет с пульсацией (AnimationController 600ms, loop)
- Фон подцели → лёгкое свечение `c.accent.withOpacity(0.12)`
- При уходе из зоны → анимация плавно гасится (200ms)

Пульсация реализуется через `AnimationController` + `CurvedAnimation(curve: Curves.easeInOut)` на `borderWidth` (от 2px до 3px) и `opacity` фонового свечения (от 0.08 до 0.18).

## Открытые вопросы

1. **Пульсация анимации** — один `AnimationController` на весь экран (переключается на активный таргет) или отдельный на каждый узел? Рекомендация: один контроллер на экране, сбрасываемый при смене `_hoverTargetId`.
2. **Блокировка зума** — блокировать ли `scaleEnabled` во время drag или оставить зум доступным? Рекомендация: блокировать оба, чтобы случайный pinch не масштабировал холст пока элемент в руках.
3. **Подцели как цель reparent** — задачи могут менять родительскую подцель (`_tryReparentTask`). А подцели могут менять родительскую подцель (`parentSubGoalId`)? Если да — нужна ли такая же логика для них? (Судя по коду — да, подцели тоже вложены).
4. **Привязанные привычки и контрольные точки** — для них reparent не предусмотрен (они привязаны к цели/подцели жёстко), поэтому подсветка цели при их перетаскивании не нужна — только haptic на start/end.
